## Evening Session — Cron Timeout Root Cause Found

### The Real Bug
Ken pushed back on my "scheduler hardcoded 120s" theory. He was right.

**Root cause:** The jobs were created on Apr 25 with `payload.timeoutSeconds: 120`. That's it. Not a scheduler bug, not an exec limit, not a gateway timeout. We literally gave them 2 minutes.

Evidence from `jobs.json.bak.2026-04-25`:
```json
{
  "name": "github-backup",
  "payload_timeout": 120,
  "root_timeout": null
}
```

Every failing job had the same 120s payload timeout. Research jobs worked because `web_search` is a fast API call (1-2s). Jobs using `exec`/`read`/`write` need time for git, file listing, memory reads, etc.

### Source Code Inspection
Read compiled OpenClaw JS to verify timeout behavior:
- `src/cron/service/timeout-policy.ts`: Default is 10 minutes (600s), agent turn safety is 1 hour (3600s)
- `resolveCronJobTimeoutMs()`: Reads `job.payload.timeoutSeconds` × 1000
- `executeJobCoreWithTimeout()`: Uses `Promise.race([jobPromise, timeoutPromise])`
- **The 120s came from our own config, not the scheduler**

### Fix Applied (Apr 26 ~19:00)
- Bulk-changed all 17 jobs' `payload.timeoutSeconds` to 300
- Also set root-level `timeoutSeconds` to 300 for safety
- Cleaned up job messages to reduce agent overhead:
  - `github-backup`: Removed nested `timeout 240` wrapper
  - `weekly-workspace-cleanup`: Added explicit paths, simplified instructions
  - `evening-accomplishment-log`: Removed `$(date +%m-%d)` shell variable the agent had to interpret

### Verification
- **Manual git push:** Completed in 44s (code 0) — git is fast, timeout was the blocker
- **Next tests:** `github-backup` at 23:00 tonight, `weekly-workspace-cleanup` next Sunday 19:00

### Ken Was Right
"K2p5 is more than strong enough to commit to git." — Correct. The issue was never model capability. It was timeout configuration from job creation day.

## DGX Spark Deep Research Completed

**File:** `memory/research/2026-04-26_dgx-spark-deep-research.md` (20,146 bytes)

### Key Findings

**Performance (confirmed real benchmarks):**
- GPT-OSS 20B (MXFP4): 83 tok/s generation, 4,500 tok/s prefill
- GPT-OSS 120B (MXFP4): 59 tok/s generation, 2,400 tok/s prefill
- Qwen3-Coder-30B (Q8_0): 61 tok/s generation
- Llama3.1 8B (Q4): 38 tok/s generation
- Llama3.1 70B (Q4): 4.4 tok/s generation (slow but fits)

**6-Agent Memory Budget (128GB):**
- Executive (120B): ~65GB
- 5 specialists (7B-14B): ~30-40GB
- Services (DB, vector, API): ~15-20GB
- **Total: ~110-125GB — tight but feasible**
- Better approach: Time-slice the 120B model, keep specialists resident

**Thermals/Noise (24/7 concern):**
- Founders Edition peaks: 87-88°C CPU, 80-82°C GPU under sustained load
- **Fan "screams" under load** — multiple reviews confirm
- ASUS GX10 has dual-fan design, may run cooler/quieter
- Power draw: ~69-76W during inference (not 240W constantly)
- At 100W average: ~€0.50/day electricity in Ireland

**Software Maturity:**
- October 2025 launch was rough (thermal throttling, reboots, ~100W cap)
- **January 2026 update: up to 2.5x performance improvement**
- February 2026 update: hot-plug, multi-monitor, Bluetooth audio
- 30+ official NVIDIA playbooks now available

**Browser Automation on DGX OS:**
- DGX OS = Ubuntu 24.04 (apt packages, NOT snap)
- **OpenClaw Issue #18123 specifically cites snap Chromium** — may NOT apply
- `apt install chromium-browser` works on ARM64
- xdotool and Xvfb available via apt on all architectures
- **Needs hands-on verification but risk is lower than initially thought**

**OpenClaw on ARM64:**
- Official Docker image `main-arm64` exists
- Raspberry Pi 5 confirmed working
- Use Docker, not npm install (avoids Issue #23987 opus compilation)

**Best Value:** ASUS Ascent GX10 at ~$3,570 (~€3,280) — same GB10 chip, $1,100 cheaper than DGX Spark, dual-fan cooling

**Verdict for 24/7 multi-agent:** Technically viable. 128GB unified memory is the enabler. Noise is the biggest practical concern for bedroom use. Browser automation risk is lower than initially assessed (DGX OS uses apt, not snap).
