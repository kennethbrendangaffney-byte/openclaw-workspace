## 🔴 Active Threats (April 2026)

### Ransomware Landscape
- **Elite Enterprise ransomware** (CYFIRMA, April 2026): High-impact Windows-targeting ransomware using AES-256 + RSA-4096 hybrid encryption. Leaves filenames unchanged to evade detection. Deletes Volume Shadow Copies, modifies boot sectors, demands 227 BTC (~$20M+). No negotiation possible — automatic decryption claimed upon payment (unverified).
- **LockBit 5.0**: Now targeting Windows, Linux, and ESXi. Ransomware-as-a-Service (RaaS) model continues to evolve.
- **Gentlemen RaaS**: New C-based locker hitting Windows, Linux, and ESXi. Fast-growing operation.
- **Payouts King ransomware**: Uses QEMU VMs to bypass endpoint security. Novel evasion technique — runs payload inside virtual machine to hide from AV/EDR.

### Linux-Specific Threats
- **GoGra backdoor (Linux variant)**: State-backed espionage group "Harvester" now targeting Linux. Uses Microsoft Graph API + Outlook for stealthy C2 communication. Hardcoded Azure AD credentials, checks Outlook inbox every 2 seconds for commands disguised as emails with "Input" subject lines. Deletes emails after execution to reduce forensics. Initial access via ELF binaries disguised as PDFs.
- **Implication**: Linux is no longer "safe by obscurity." State actors are actively developing Linux tooling.

### Key Takeaway
Ransomware is increasingly multi-platform (Windows/Linux/ESXi), using novel evasion (VM-based payload delivery), and state actors are expanding to Linux. Self-hosted infrastructure is a target.

## 🛡️ Linux Hardening for Self-Hosters (2026)

### Critical Basics (Do These First)
1. **SSH lockdown**: Key-based auth only, disable password auth, disable root login, change default port (or use port knocking)
2. **Firewall**: UFW or nftables — default deny, allow only necessary ports
3. **Fail2ban**: Brute-force protection for SSH and exposed services
4. **Automatic updates**: unattended-upgrades for security patches
5. **Non-root execution**: Run all services as unprivileged users
6. **2FA for SSH**: TOTP (Google Authenticator, YubiKey) — especially critical for internet-facing servers

### Self-Hosting Specifics
- **Reverse proxy**: Caddy or Traefik with automatic HTTPS (Let's Encrypt)
- **Container isolation**: Docker/Podman with user namespaces, seccomp profiles, AppArmor/SELinux
- **Network segmentation**: VLANs or separate Docker networks per service
- **Secrets management**: Don't hardcode credentials in configs — use Docker secrets, environment files with restricted permissions (600), or HashiCorp Vault for larger setups
- **Backup strategy**: 3-2-1 rule — 3 copies, 2 different media, 1 offsite. Encrypted backups tested regularly.
- **Monitoring**: Log aggregation (Loki/Fluentd), intrusion detection (AIDE, OSSEC), resource monitoring (Prometheus/Grafana)

### Advanced Hardening
- **Kernel hardening**: sysctl tweaks (disable IPv6 if unused, enable SYN cookies, disable source routing)
- **GRUB password**: Prevent single-user mode access
- **Filesystem encryption**: LUKS for data at rest
- **Audit framework**: auditd for compliance and forensics
- **Application sandboxing**: Firejail, bubblewrap, or systemd sandboxing options

## 👤 Personal OPSEC (2026)

### Core Principles
1. **Identify sensitive info**: What could compromise you if exposed? (Addresses, schedules, health data, financials, family info)
2. **Analyze threats**: Who might want this? (Data brokers, scammers, stalkers, state actors — threat model varies)
3. **Find vulnerabilities**: Where is this info exposed? (Social media, public records, breached databases)
4. **Implement countermeasures**: Reduce exposure, add friction for attackers
5. **Continuous monitoring**: Regular audits of digital footprint

### Practical Steps
- **Data broker opt-out**: Remove yourself from Spokeo, WhitePages, BeenVerified, etc. (tedious but effective)
- **Social media hygiene**: Minimize real info, disable location tagging, review privacy settings quarterly
- **Email aliasing**: Use unique emails per service (SimpleLogin, AnonAddy, or AliasVault) — limits breach cascade
- **Phone number privacy**: Google Voice, MySudo, or similar for non-essential services
- **Payment masking**: Privacy.com virtual cards, prepaid cards for subscriptions
- **Browser fingerprinting**: Use Firefox with resistFingerprinting, or Tor Browser for anonymity
- **Search habits**: DuckDuckGo, Startpage, or self-hosted SearXNG — avoid Google for sensitive queries
- **Device hygiene**: Separate devices/accounts for different risk levels (work, personal, high-risk browsing)

### High-Risk Additions
- **Post-quantum prep**: Begin migrating to quantum-resistant encryption where available (still early, but signal that crypto agility matters)
- **On-device privacy**: Process sensitive data locally before syncing (Ken's local AI setup aligns with this)
- **Travel OPSEC**: Separate travel devices, minimal data crossing borders, encrypted containers
