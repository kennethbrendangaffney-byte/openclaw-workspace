## Executive Summary

Gemma 4 is Google's most capable open model family to date, built from the same research and technology as Gemini 3. It delivers **unprecedented intelligence-per-parameter** with frontier-level reasoning, native multimodal capabilities (text, image, audio, video), and agentic workflow support.

**Key Achievement:** Gemma 4-31B ranks as the **#3 open model** on Arena AI text leaderboard, while the 26B MoE secures **#6 spot** — outperforming models 20x their size.

**Strategic Position:** Apache 2.0 licensed, commercially permissive, with day-0 support across virtually every inference framework (llama.cpp, Ollama, vLLM, MLX, transformers.js, etc.).

## Architecture Innovations

### 1. Per-Layer Embeddings (PLE)
**What it is:** A second, lower-dimensional embedding pathway that feeds into every decoder layer alongside the main residual stream.

**How it works:**
- Each token gets a dedicated per-layer vector
- Combines token-identity (embedding lookup) + context-aware (learned projection)
- Modulates hidden states via lightweight residual block
- Computed before multimodal soft tokens merge

**Benefit:** Each layer receives token-specific information only when relevant, rather than forcing everything into a single upfront embedding. Adds specialization at modest parameter cost.

### 2. Shared KV Cache
**What it is:** Last N layers reuse key-value states from earlier layers instead of computing their own.

**How it works:**
- Last `num_kv_shared_layers` don't compute K/V projections
- Reuse K/V from last non-shared layer of same attention type (sliding or full)
- Minimal quality impact, significant memory/compute savings

**Benefit:** Critical for long-context generation and on-device inference. Makes 128K-256K context practical.

### 3. Alternating Attention Patterns
- **Sliding Window:** Local attention (512 tokens for small models, 1024 for large)
- **Global Full-Context:** Full attention layers interleaved
- **Dual RoPE:** Standard RoPE for sliding, pruned RoPE for global (enables longer context)

### 4. Vision Encoder Improvements
- **Variable Aspect Ratios:** Preserves original image proportions (no forced square cropping)
- **Configurable Token Budgets:** 70, 140, 280, 560, 1120 tokens per image
- **Learned 2D Positions + Multidimensional RoPE**
- **Trade-off Control:** Speed vs memory vs quality per use case

### 5. Audio Encoder (E2B/E4B only)
- **Architecture:** USM-style conformer (same as Gemma-3n)
- **Capabilities:** Speech recognition, audio question answering, transcription
- **Training:** Speech-focused (music/non-speech not in training data)

## Multimodal Capabilities

### Native JSON Object Detection
Gemma 4 outputs bounding boxes natively in JSON format without grammar constraints:
```json
[{"box_2d": [171, 75, 245, 308], "label": "view recipe element"}]
```
- Coordinates relative to 1000x1000 image space
- Works across all model sizes

### Video Understanding
- All models understand video content
- E2B/E4B: Video + audio analysis
- 26B/31B: Video without audio
- Strong performance even without explicit video post-training

### Audio Processing (E2B/E4B)
- **Speech Recognition:** Accurate transcription
- **Audio QA:** Answer questions about speech content
- **Limitation:** Music and non-speech sounds not in training data

### Multimodal Function Calling
- Native tool use with vision input
- Example: Identify city from image → call weather API
- Works across all sizes with thinking mode enabled

## Local Deployment Options

### 1. Ollama (Easiest)
```bash
# Available models: gemma4:2b, gemma4:4b, gemma4:26b, gemma4:31b
ollama run gemma4:4b
```
- Day-0 support
- Simplest setup for immediate use

### 2. llama.cpp (Maximum Control)
```bash
# Start OpenAI-compatible server
llama-server -hf ggml-org/gemma-4-E2B-it-GGUF

# Or quantized versions
llama-server -hf ggml-org/gemma-4-26b-a4b-it-GGUF:Q4_K_M
```
- Full GGUF quantization control
- CPU, CUDA, Metal backends
- Compatible with LM Studio, Jan, coding agents

### 3. Hugging Face Transformers
```python
from transformers import pipeline
pipe = pipeline("any-to-any", model="google/gemma-4-e2b-it")

# Multimodal input
messages = [{
    "role": "user",
    "content": [
        {"type": "image", "image": "photo.jpg"},
        {"type": "text", "text": "Describe this image"}
    ]
}]
output = pipe(messages, max_new_tokens=100)
```

### 4. MLX (Apple Silicon)
```bash
pip install mlx-vlm
mlx_vlm.generate \
  --model google/gemma-4-E4B-it \
  --image photo.jpg \
  --prompt "Describe this image"

# With TurboQuant (4x memory reduction)
mlx_vlm.generate \
  --model "mlx-community/gemma-4-26b-a4b-it-4bit" \
  --kv-bits 3.5 \
  --kv-quant-scheme turboquant
```

### 5. mistral.rs (Rust-native)
```bash
# Multimodal with tool calling
curl --proto '=https' --tlsv1.2 -sSf \
  https://raw.githubusercontent.com/EricLBuehler/mistral.rs/master/install.sh | sh

mistralrs run -m google/gemma-4-E4B-it --isq 8 --image image.png -i "Describe this"
mistralrs run -m google/gemma-4-E4B-it --isq 8 --audio audio.mp3 -i "Transcribe this"
```

### 6. transformers.js (Browser)
- Run Gemma 4 directly in browser via WebGPU
- Demo: https://huggingface.co/spaces/webml-community/Gemma-4-WebGPU
- ONNX checkpoints available

## GGUF Quantization Guide

| Quant | Use Case | Quality | Speed | Size |
|-------|----------|---------|-------|------|
| **Q4_K_M** | Balanced | Good | Fast | ~16GB (31B) |
| **Q5_K_M** | Quality-priority | Better | Medium | ~19GB (31B) |
| **Q6_K** | Near-lossless | Excellent | Slower | ~23GB (31B) |
| **Q8_0** | Maximum quality | Best | Slowest | ~31GB (31B) |

**Recommendations:**
- **24GB VRAM:** Q4_K_M for 31B, Q6_K for 26B-A4B
- **16GB VRAM:** Q4_K_M for 26B-A4B, Q6_K for E4B
- **8GB VRAM:** E4B Q4_K_M or E2B Q8_0
- **Mobile/Edge:** E2B int8 or E4B Q4_0

**Download:** https://huggingface.co/collections/ggml-org/gemma-4

## Comparison with Qwen3.6

| Aspect | Gemma 4 | Qwen3.6 |
|--------|---------|---------|
| **License** | Apache 2.0 ✅ | Apache 2.0 (35B-A3B only) |
| **Top Model Open** | Yes (31B dense) | No (Plus is API-only) |
| **MoE Variant** | 26B-A4B (4B active) | 35B-A3B (3B active) |
| **Coding (SWE-bench)** | 52.0 (31B) | 73.4 (35B-A3B) |
| **Terminal Tasks** | 42.9 (31B) | 51.5 (35B-A3B) |
| **Arena Ranking** | #3 (31B) | Not ranked (API-only) |
| **Ecosystem** | Day-0 everywhere | Limited (no official GGUF) |
| **Audio Input** | Yes (E2B/E4B) | No |
| **Mobile/Edge** | Excellent (E2B/E4B) | Limited |
| **Multimodal** | Excellent | Excellent |
| **Agent Support** | Native function calling | Native function calling |

**Verdict:**
- **Choose Gemma 4 if:** You want maximum ecosystem support, mobile/edge deployment, audio capabilities, or easy local setup via Ollama/llama.cpp
- **Choose Qwen3.6 if:** You need the absolute best coding/agentic performance and can use API or accept limited local framework support

## Limitations & Considerations

1. **Audio Limitations:** E2B/E4B audio only trained on speech; music/non-speech handling is poor
2. **Video Training:** Not explicitly post-trained on video, but generalizes well
3. **Large Model Requirements:** 31B needs significant VRAM even quantized
4. **MoE Complexity:** 26B-A4B requires framework MoE support (most now have it)
5. **GGUF Quality:** Lower quantization levels may impact multimodal performance more than text
