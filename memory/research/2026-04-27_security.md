## 2. Linux Hardening Techniques (2026 Best Practices)

### OS Hardening Fundamentals

**Minimal Install Philosophy**
- Install less, run fewer services, restrict permissions
- Every extra package = one more thing to patch
- Disable unused services completely
- Apply disciplined LAMP stack hardening for web services

**Service Configuration**
- Bind services only to interfaces they need (not 0.0.0.0 by default)
- Run services with least privilege (never root if unnecessary)
- Fix file permissions: eliminate world-writable directories, sloppy config files

**Mandatory Access Control (MAC)**
- **SELinux** or **AppArmor** are essential despite tuning overhead
- Can contain damage even when services are exploited
- Real-world example: Finance environment where attackers compromised web service but couldn't pivot due to SELinux policies

### Patching and Vulnerability Management

**The Pipeline Approach:**
1. Discover issues continuously
2. Prioritize based on exploitability and impact
3. Test updates where possible
4. Deploy in controlled waves
5. Verify and monitor

**Critical security fixes need hours, not weeks**
- Use vulnerability scanners for consistency (humans forget)
- Monitor for configuration drift
- Track third-party dependencies (large dependency graphs = hidden risks)

### Network Segmentation & Lateral Movement Prevention

**Segmentation Strategy:**
- Public-facing services in DMZ
- App tiers in protected internal zones
- Databases in heavily restricted data networks
- Management interfaces isolated and locked down

**Host Firewalls:**
- Default-deny stance with explicit allow rules only
- Reduces blast radius when perimeter assumptions fail

**Remote Access Hardening:**
- MFA is non-negotiable
- Certificate-based authentication where possible
- Restrict source IPs
- Short-lived credentials
- Strong logging

### Access Control & Privileged Account Security

**Core Principles:**
- MFA is table stakes
- Least privilege: limit who has admin rights, when they're granted, log all sessions
- Use just-in-time elevation instead of permanent admin access
- Modern Linux security = "who can do what, when, and why"

## 3. Self-Hosted Privacy & Security Tools (2026)

### Essential Security Stack for Self-Hosters

| Layer | Tool | Purpose |
|-------|------|---------|
| **Network Perimeter** | WG-Easy (WireGuard) | Keeps admin services off public internet |
| **Intrusion Prevention** | CrowdSec + Fail2Ban | Blocks brute force, scans, malicious IPs |
| **Secrets Management** | HashiCorp Vault / OpenBao | Centralized encrypted credential storage |
| **Password Management** | Vaultwarden | Self-hosted Bitwarden-compatible password vault |
| **SSL/TLS** | Let's Encrypt | Automated certificate management |

### Detailed Tool Breakdown

#### 1. CrowdSec (Collaborative Intrusion Prevention)
- **60x faster than Fail2Ban** (written in Go)
- IPv6 compatible out of the box
- Built-in Web Application Firewall (WAF)
- Community blocklist updated continuously by thousands of nodes
- Shares threat intelligence across the network
- Works with Docker, Kubernetes, Linux, OPNsense

#### 2. HashiCorp Vault / OpenBao (Secrets Management)
- Centralized encrypted store for API keys, passwords, certificates
- Dynamic secrets (auto-generated, auto-expired credentials)
- Encryption as a service for applications
- **OpenBao** is the Linux Foundation fork (MIT license) if Vault's licensing is a concern

#### 3. WG-Easy (WireGuard VPN)
- Clean web UI for WireGuard management
- One-click client creation with QR codes
- Access self-hosted services securely from anywhere
- Keep admin panels off public internet
- Lightweight, minimal resource usage

#### 4. Vaultwarden (Password Management)
- Lightweight Bitwarden implementation (~50MB RAM)
- Full Bitwarden client compatibility
- AES-256 encryption with Argon2 key derivation
- Runs on Raspberry Pi
- Supports organizations, shared vaults, 2FA

#### 5. Fail2Ban (Classic Defense)
- Zero learning curve, battle-tested
- Custom jail configurations for any service
- Lightweight, no external dependencies
- Perfect complement to CrowdSec (defense in depth)

## 4. Personal OPSEC Recommendations

### Immediate Actions (High Impact, Low Effort)

1. **Install CrowdSec** on every public-facing server
2. **Set up WG-Easy** to access admin panels via VPN only
3. **Deploy Vaultwarden** and migrate off browser-saved passwords
4. **Enable MFA** everywhere possible
5. **Review SSH configuration:** disable password auth, use keys only, change default port

### Defense in Depth Strategy

No single tool is enough. Layer your defenses:
- Network perimeter (VPN/WAF)
- Host-level protection (CrowdSec/Fail2Ban)
- Application security (secrets management)
- Credential hygiene (password manager)
- Transport security (TLS/SSL)

### Key Statistics to Remember

- **Weak secrets management contributed to 22% of cloud data breaches in 2025**
- **Risky AI prompts increased by 97% in 2025**
- **40% of Model Context Protocols (MCPs) were vulnerable**
- **597+ Linux vulnerabilities patched in March 2026 alone**

### Self-Hosting Security Mindset

> "Every service you spin up is another attack surface. And if you're not actively protecting your infrastructure, you're essentially leaving the front door wide open."

Self-hosters are personally responsible for every security decision. The tools above are open-source and free—hosting the entire security stack costs under $50/month in infrastructure.
