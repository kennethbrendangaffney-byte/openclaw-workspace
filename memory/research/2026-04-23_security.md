## 1. Current Threat Landscape (2026)

### Key Trends

- **Supply Chain Attacks Are Surging** — March/April 2026 saw significant increases in software supply chain compromises. The Axios NPM package compromise was a notable incident, attributed to North Korean actors. Attackers are poisoning legitimate libraries and build tools, spreading malware through normal update channels.
- **Ransomware Targeting Servers** — Server-side ransomware is increasingly lucrative. Encrypting a database or production filesystem stops operations cold. Linux servers are firmly in the crosshairs because that’s where business-critical assets live.
- **AI-Assisted Reconnaissance** — Attackers now use AI to speed up reconnaissance and vulnerability discovery. Exploitation cycles run in hours, not weeks.
- **Self-Reinforcing Cybercrime Economy** — Researchers describe a cascading cycle where supply chain breaches link identity abuse, SaaS compromise, and ransomware into a continuous loop.
- **Credential Leaks & Misconfigurations** — Even when patching is current, leaked credentials and misconfigurations remain common entry points.

### What This Means for Self-Hosted Setups

Your home server or VPS is being scanned within minutes of going online. Automated bots probe for open ports, weak passwords, and unpatched services constantly. The bar for "secure enough" has risen — "I have a firewall" is no longer a strategy.

## 2. Linux Hardening — Practical Checklist

### SSH Hardening (Critical — Do This First)

SSH is the front door. If it’s compromised, everything else is irrelevant.

**Key steps:**
- Create a non-root user with sudo privileges, disable root login
- Use SSH key authentication only (`ed25519` keys recommended)
- Disable password authentication entirely
- Restrict to specific users: `AllowUsers youruser`
- Use strong algorithms only:
  - `KexAlgorithms`: curve25519-sha256 variants
  - `Ciphers`: chacha20-poly1305, aes256-gcm
  - `MACs`: hmac-sha2-512-etm, umac-128-etm
- Set idle timeouts: `ClientAliveInterval 300`, `ClientAliveCountMax 2`
- Consider non-standard port (reduces log noise, not real security)
- **Always** validate config with `sshd -t` before restarting
- Keep current session open, test new connection in separate terminal

**Reference:** [ZeonEdge SSH Hardening Guide 2026](https://zeonedge.com/blog/ssh-hardening-2026-complete-guide-linux-server)

### Firewall Configuration

- **Default deny** — block all incoming, allow only what’s needed
- UFW (Ubuntu/Debian) or nftables (modern, flexible)
- Allow SSH first, then enable firewall
- Rate-limit SSH: `ufw limit ssh`
- Allow only specific IPs for sensitive services (databases, admin panels)

### Brute Force Protection

- **Fail2Ban** — monitors logs, auto-bans malicious IPs
- Set `maxretry = 3`, `bantime = 3600` (1 hour)
- Progressive banning for repeat offenders (24-hour bans)
- Whitelist your own IP to avoid lockouts

### System Updates & Patching

- Unpatched software is the single largest attack surface
- Enable automatic security updates:
  - Debian/Ubuntu: `unattended-upgrades`
  - RHEL: `dnf-automatic`
- Treat vulnerability management as a pipeline: discover → prioritize → test → deploy → verify

### Disable Unnecessary Services

- Every running service is a potential attack surface
- List running services: `systemctl list-units --type=service --state=running`
- List listening ports: `ss -tlnp`
- Disable unused services: printing, Bluetooth, mDNS, RPC/NFS (if not needed)

### File Permissions

- Set restrictive permissions on sensitive files:
  - `chmod 600 /etc/ssh/sshd_config`
  - `chmod 700 /root`
  - `chmod 400` for config files with secrets
- Check for world-writable files: `find / -xdev -type f -perm -0002 -ls`
- Check for unexpected SUID/SGID binaries: `find / -xdev \( -perm -4000 -o -perm -2000 \) -type f -ls`

### Mandatory Access Control (MAC)

- **SELinux** (RHEL-family) or **AppArmor** (Debian/Ubuntu)
- Don’t disable these — learn to work with them
- MAC can contain damage even when a service is exploited
- Real-world example: attackers compromised a web service but couldn’t pivot because SELinux restricted access to sensitive data

### DNS Security

- Use trusted resolvers: Cloudflare (`1.1.1.1`) or Quad9 (`9.9.9.9` — malware filtering)
- Consider running a local recursive resolver (Unbound) for privacy and performance
- Prevent `resolv.conf` from being overwritten: `chattr +i /etc/resolv.conf`

### Kernel & Network Hardening

- Minimal installs — fewer packages = smaller attack surface
- Bind services to specific interfaces, not `0.0.0.0` unless necessary
- Run services as non-root users
- Network segmentation: DMZ for public-facing, restricted zones for databases
- Host-based firewalls with default-deny stance

## 3. Self-Hosted Privacy & Security Tools

### Password Management

- **Vaultwarden** — self-hosted Bitwarden-compatible password manager (Rust rewrite, lighter resource usage)
- **AliasVault** — privacy-first password manager with built-in email aliasing, fully encrypted and self-hostable

### Secure Access & VPN

- **Octelium** — next-gen FOSS self-hosted zero trust secure access platform. Can operate as remote access VPN, ZTNA platform, API/AI/MCP gateway, PaaS, ngrok-alternative
- **SwizGuard** — self-hosted "Stealth VPN" forked from xray-core and WireGuard. Makes traffic look like normal TLS while hiding an encrypted WireGuard tunnel inside

### Secure Communication

- **XMail** — self-hosted, federated, zero-knowledge encrypted mail system in a single PHP file. AES-256-GCM encryption, PBKDF2 key derivation. Server never sees message passwords

### AI Security Proxy

- **AISecurityGateway** — self-hosted AI security proxy. Redacts PII, blocks prompt injection, routes to any LLM provider. OpenAI-compatible API

### General Privacy

- **FlowGuard** — privacy-first traffic filtering proxy (currently in private beta)
- Local DNS resolver (Unbound) + DNS-over-HTTPS/TLS for encrypted DNS queries

## 4. Personal OPSEC Recommendations

### For Self-Hosted/Home Lab Setups

1. **Assume you’re being scanned constantly** — every public IP is probed within minutes
2. **SSH keys only** — never passwords for remote access
3. **Firewall first** — default deny, explicit allow rules only
4. **Automated updates** — enable unattended security updates
5. **Monitor logs** — check `/var/log/auth.log` regularly, set up alerts for anomalies
6. **Backup encryption** — encrypt backups before sending anywhere off-site
7. **Segment your network** — IoT devices on separate VLAN, management interfaces isolated
8. **Use a password manager** — unique, strong passwords for every service
9. **Email aliasing** — use unique emails per service (AliasVault or SimpleLogin) to reduce breach impact
10. **Minimal exposure** — don’t expose services to the internet unless absolutely necessary. Use VPN or reverse proxy with auth

### Supply Chain Awareness

- Verify package signatures before installing
- Pin dependency versions, don’t blindly `apt upgrade` without checking changelogs for critical services
- Use container images from trusted sources, scan for vulnerabilities
- Be wary of "install this curl | bash script" instructions

### Monitoring & Response

- Set up log aggregation (even simple `logwatch` or `fail2ban` reports)
- Know what "normal" looks like so you can spot anomalies
- Have a recovery plan: backups tested, restore procedure documented
