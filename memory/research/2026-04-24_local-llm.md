### 2. llama.cpp - The Engine Powering Local AI

**Current state (April 2026):**
- 100K+ stars on GitHub, maintained under ggml-org
- The de-facto runtime for local open-weight models
- Continuous releases (b8000+ series) with active development

**Recent major developments:**
- **Gemma 4 audio support** (April 2026) - multimodal audio processing
- **Speculative decoding** - up to 3x speed improvement
- **Low-level CPU optimizations** - hand-optimized dot product kernels for q1_0 quantization (PR #21636 by pl752)
  - Uses AVX2/AVX-512 vector instructions
  - Closes the gap between local inference and cloud GPU performance
  - Memory bandwidth is the real constraint, not raw compute
- **RDNA4-specific optimizations** for AMD GPUs
- **PowerPC FP16 MMA path** for Q4_0 (b8099)
- **Hexagon DSP support** for Qualcomm devices

**CPU inference reality:**
- Modern CPU + DDR5: 2-5 tok/s (batch processing)
- With Q4 quantization: 3-6 tok/s (non-interactive)
- Optimized kernels can deliver throughput improvements no CPU upgrade can match
- 3-7B models with Q4 quantization are viable for CPU-only deployment

### 3. Quantization Techniques - The Full Picture

**GGUF Format (llama.cpp standard):**
Single-file format with embedded metadata, native support in llama.cpp and Ollama.

**Quantization Levels (7B model reference):**

| Format | Effective Bits | Size (7B) | Perplexity Δ | Notes |
|--------|---------------|-----------|--------------|-------|
| Q8_0 | ~8-bit | ~6.7 GB | +0.0004 | Near-lossless; safe baseline |
| Q6_K | ~6.0 | ~5.15 GB | +0.0044 | "Almost lossless" |
| Q5_K_M | ~5.1 | ~4.45 GB | +0.0142 | Recommended high-quality 5-bit |
| **Q4_K_M** | **~4.5** | **~3.80 GB** | **+0.0535** | **Most popular default (sweet spot)** |
| Q4_K_S | ~4.3 | ~3.56 GB | +0.1149 | Smaller 4-bit, more loss |
| Q3_K_M | ~3.3 | ~3.06 GB | +0.2437 | Balanced 3-bit |
| Q2_K | ~2.5 | ~2.67 GB | +0.8698 | Extreme compression, not recommended |

**I-Quant formats (aggressive compression):**
- IQ4_XS: ~4.25 bits, best quality/size at 4-bit, slightly slower decode
- IQ3_M: ~3.6 bits, maximum compression with quality preservation

**K-Quants methodology:**
- Block-wise quantization with per-group scales
- Two-level block quantization (small blocks → super-blocks)
- Double-quantized scales for better quality-per-bit
- Naming: Q4_K_M = 4-bit, K-quants, Medium metadata density

**TurboQuant (experimental in llama.cpp):**
- Prioritizes inference speed over traditional GGUF schemes
- Unlike Q4_K_M or Q5_K_M that balance compression and quality
- Speed-first approach for latency-sensitive applications

**Other quantization methods:**
- **FP8**: Gold standard for GPU (Hopper+), ~99% quality, 33% faster than FP16
- **AWQ**: Activation-aware, preserves salient weights (top 1%), ~95% quality
- **GPTQ**: GPU-optimized, excellent throughput, supports 2/3/4/8-bit
- **INT8/INT4**: Wide compatibility, 97-99% / 90-95% quality retention

### 4. BitNet - Microsoft's 1.58-bit Revolution

**BitNet b1.58 2B4T** (released March 2026):
- First open-source, natively trained 1-bit LLM
- 2B parameters, trained on 4T tokens
- Weights constrained to {-1, 0, +1} (1.58 bits effective)
- Performance comparable to full-precision models of similar size
- Substantial advantages: memory, energy, latency

**bitnet.cpp inference framework:**
- Official Microsoft inference framework for 1-bit LLMs
- 38K+ stars on GitHub
- Runs on standard CPUs (no GPU required)
- Python (50%) + C++ (46%) implementation

**Current limitations:**
- Largest working model: 2B parameters
- 8B models exist but require kernel rebuilds (TL1 format)
- I2_S conversions have issues for larger models
- Context: 4096 tokens (Llama3.2-1B architecture base)

**Performance on our system:**
- ~27 tokens/sec (8 threads, CPU)
- 1.3GB download, 1.1GB RAM at runtime
- Most aggressive quantization available

4. **BitNet 2B** (I2_S) - ~1.1GB RAM
   - Most memory-efficient
   - Fastest inference (~27 tok/s)
   - Good for simple tasks
   - `~/BitNet/build/bin/llama-cli -m models/BitNet-b1.58-2B-4T/ggml-model-i2_s.gguf`

**Quantization strategy:**
- Default: Q4_K_M (best quality/size balance)
- Memory constrained: Q4_K_S or IQ4_XS
- Maximum quality: Q5_K_M or Q6_K
- Extreme compression: Q3_K_M (acceptable quality loss)

**llama.cpp optimization tips:**
- Use latest build for CPU optimizations
- Enable AVX2/AVX-512 if available
- Consider TurboQuant for speed-critical applications
- Speculative decoding for up to 3x throughput

### 6. Emerging Trends to Watch

- **quant.cpp** (quantumaikr/synapticore-io): Pure C, zero dependencies, 7x longer context via lossless KV cache compression
- **Delta KV compression**: 4x longer context in embeddable inference
- **Continuous batching**: Up to 23x throughput improvement
