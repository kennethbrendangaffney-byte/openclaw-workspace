# Local LLM Research Summary — April 21, 2026

## Executive Summary

Local LLM inference has matured significantly in 2026. The ecosystem is dominated by three pillars: **Ollama** (developer-friendly wrapper), **llama.cpp** (foundational inference engine), and **BitNet** (extreme quantization for CPU-only deployment). Open-weight models now deliver 70-85% of frontier model quality at zero marginal cost per request.

---

## Key Platforms

### Ollama (v0.17+)
- **Status:** The default choice for developers — 166k GitHub stars, 52M monthly downloads in Q1 2026
- **Recent updates (Feb 2026):**
  - New inference engine: up to 40% faster prompt processing, 18% faster token generation on NVIDIA
  - 8-bit KV cache quantization — halves memory footprint for long contexts
  - `ollama launch` command — one-command setup for coding tools (Claude Code, Cline, OpenClaw, etc.)
  - Native MLX runner on Apple Silicon
  - Anthropic Messages API compatibility (in addition to OpenAI API)
  - Auto-context sizing on macOS/Windows based on available memory
- **Best for:** Rapid prototyping, RAG pipelines, agentic workflows, drop-in API replacement

### llama.cpp
- **Status:** The foundational C/C++ inference engine (98.6k stars) that powers most local inference
- **Key capabilities:**
  - Widest quantization support: Q1_5_K through Q8_0
  - GGUF format standard (135,000+ models on HuggingFace)
  - Multi-backend: CPU (AVX512), CUDA, Metal, Vulkan, ROCm
  - MoE (Mixture of Experts) model support
  - Distributed inference across networked machines
- **Performance:** ~95 tok/s on Llama 3.1 8B Q4_K_M (MacBook Pro M3 Pro)
- **Best for:** Maximum throughput, ultra-low-bit quantization, production latency-sensitive apps

### BitNet (Microsoft Research)
- **Status:** Official inference framework for 1-bit (ternary) LLMs — trending on GitHub
- **Key innovation:** 1.58-bit ternary quantization (weights: -1, 0, +1)
- **Performance claims:**
  - 1.37x–5.07x speedup on ARM CPUs
  - 2.37x–6.17x speedup on x86 CPUs
  - 55–82% energy reduction
  - 16x less memory than full precision
- **Practical reality:** 100B models on CPU are experimental; 2B models run at ~28 tok/s using 0.5GB RAM
- **Best for:** Edge deployment, CPU-only hardware, IoT/battery-powered devices, extreme RAM constraints

## Quantization Landscape

| Format | Bits | Use Case | Quality Impact |
|--------|------|----------|----------------|
| Q8_0 | 8-bit | Maximum quality, large VRAM | Minimal |
| **Q4_K_M** | 4-bit | **Sweet spot for most users** | Slight |
| Q3_K_M | 3-bit | Tight VRAM constraints | Moderate |
| Q2_K | 2-bit | Extreme constraints | Noticeable |
| **BitNet b1.58** | 1.58-bit | CPU-only, edge, IoT | Research-grade |

**Rule of thumb:** Q4_K_M gives you ~25-30% of original model size with minimal quality loss. Most 7B models run comfortably in 4-8GB VRAM at this precision.

## Cost Analysis: Local vs Cloud

- **Crossover point:** ~1,000 requests/day makes local cheaper than cloud APIs
- **At 50K requests/day:** Cloud = ~$2,250/month; Local = ~$15/month electricity
- **Hardware amortization:** RTX 4090 build = $55/month over 36 months

## Recommendations for Our Setup

**Current stack:** Ubuntu, 24GB RAM, no discrete GPU, Qwen2.5:7b-q2, BitNet 2B

**Observations:**
1. Our Qwen2.5:7b-q2 at Q2_K is aggressive quantization — consider Q4_K_M if we can fit it in RAM
2. BitNet 2B at 27 tok/s is impressive for CPU-only — watch for larger BitNet models as they mature
3. Ollama 0.17's KV cache quantization could help us run longer contexts within 24GB
4. Qwen3-Coder-Next should be our next model trial for coding tasks

**Actions to consider:**
- Test Qwen3.5 7B Q4_K_M (should fit in 24GB system RAM)
- Monitor BitNet releases for larger working models (4B+ would be significant)
- Upgrade to Ollama 0.17+ for performance and KV cache improvements
- Evaluate `ollama launch` integration for coding workflows
