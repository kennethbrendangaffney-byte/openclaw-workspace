# Local LLM Research Summary - 2026-04-29

## Overview

The local LLM landscape in 2026 is defined by three converging trends: dramatically more capable small models (1B-8B), advanced quantization techniques that shrink models without destroying quality, and CPU inference frameworks that make large models accessible without GPUs. The "intelligence density" race has replaced the parameter-count arms race.

---

## Key Developments

### 1. Small Language Models (SLMs) - The New Sweet Spot

The most significant shift in 2026 is that small models now punch far above their weight. Research from Epoch AI shows compute required for equivalent capability halves every ~8 months. What needed 70B parameters in 2024 is achievable with 8B in 2026.

### 2. Quantization Techniques

Quantization remains the single most important technology for local deployment. The GGUF format (GPT-Generated Unified Format) is the universal standard.

**GGUF Quantization Levels:**

| Format | Bits/Param | 7B Size | 14B Size | Quality vs FP16 |
|--------|-----------|---------|----------|-----------------|
| FP16 | 16 | 14.0 GB | 28.0 GB | Reference |
| Q8_0 | 8.5 | 7.5 GB | 15.0 GB | ~99% |
| Q6_K | 6.6 | 5.8 GB | 11.5 GB | ~98% |
| Q5_K_M | 5.7 | 5.0 GB | 10.0 GB | ~97% |
| **Q4_K_M** | 4.8 | 4.2 GB | 8.5 GB | ~95% |
| Q3_K_M | 3.9 | 3.4 GB | 7.0 GB | ~90% (notable loss) |
| Q2_K | 3.0 | 2.6 GB | 5.5 GB | Significant degradation |

**Recommendations:**
- **Q4_K_M** is the default sweet spot for most use cases
- **Q5_K_M** or Q6_K for quality-critical applications (RAG, code, reasoning)
- **Q8_0** only when memory is abundant and near-FP16 quality is needed
- **Q3_K_M and below** only as last resort — quality drop is visible

**Advanced technique:** Use importance matrix (imatrix) quantization for better quality at lower bit rates:
```bash
./llama-imatrix -m model.gguf -f calibration_data.txt -o imatrix.dat
./llama-quantize --imatrix imatrix.dat model.gguf model-q4.gguf Q4_K_M
```

**KV Cache Quantization:** For long contexts (32K+), the KV cache can exceed weight size. llama.cpp and MLX support quantized KV cache (Q8 or Q4) which halves or quarters memory use with negligible quality cost.

### 3. llama.cpp - The Foundation

llama.cpp remains the C/C++ inference engine that everything else builds on. Created by Georgi Gerganov, it pioneered running LLMs on consumer hardware via quantization.

**CPU Optimizations:**
- AVX, AVX2, AVX512 instruction sets (x86)
- ARM NEON, ARM I8MM (smmla) for int8 matrix multiplies
- OpenBLAS integration
- Multi-threading with configurable thread counts

**GPU Backends:**
- CUDA (NVIDIA)
- HIP (AMD)
- Vulkan (cross-platform)
- SYCL (Intel)
- MUSA (Moore Threads)

**Key Features:**
- GGUF format support (fast loading, cross-framework)
- REST API server mode (OpenAI-compatible)
- Quantized KV cache
- Importance matrix quantization
- Speculative decoding for faster inference

### 4. BitNet - 1.58-Bit Ternary Quantization

Microsoft's BitNet is the most radical quantization approach, using ternary weights (-1, 0, +1) at ~1.58 bits per parameter.

**Key Facts:**
- **bitnet.cpp** is the official inference framework (forked from llama.cpp)
- Speedups: 1.37x-5.07x on ARM CPUs, 2.37x-6.17x on x86 CPUs
- Energy reduction: 55-70% on ARM, 72-82% on x86
- Can run **100B parameter models on a single CPU** at human reading speed (5-7 tok/s)
- Latest optimization (Jan 2026) adds parallel kernels with configurable tiling

**Supported Models:**
| Model | Params | x86 | ARM |
|-------|--------|-----|-----|
| BitNet-b1.58-2B-4T | 2.4B | ✅ | ✅ |
| bitnet_b1_58-large | 0.7B | ✅ | ✅ |
| bitnet_b1_58-3B | 3.3B | ✅ | ✅ |
| Llama3-8B-1.58-100B-tokens | 8.0B | ✅ | ✅ |
| Falcon3 Family | 1B-10B | ✅ | ✅ |
| Falcon-E Family | 1B-3B | ✅ | ✅ |

**Kernel formats:** I2_S (working), TL1 (limited), TL2 (limited)

**Important caveat:** BitNet requires models trained from scratch with ternary weights — you can't just quantize an existing model to 1.58-bit and expect it to work.

### 5. Hardware Tiers for Local LLMs (2026)

**Tier 1 — Entry/Laptop (8-16GB RAM):**
- Models: 3B-8B at Q4_K_M
- Speed: 15-35 tok/s
- Examples: MacBook Air M2/M3/M4 16GB, RTX 4060 8GB

**Tier 2 — Sweet Spot (24-48GB RAM):**
- Models: 13B-14B at Q5/Q6, 32B at Q4
- Speed: 25-80 tok/s
- Examples: MacBook Pro M3/M4 Pro, RTX 3090/4090 24GB, RTX 4070 Ti Super 16GB

**Tier 3 — Power User (64-128GB RAM):**
- Models: 32B at Q6/Q8, 70B at Q4_K_M
- Speed: 10-25 tok/s for 70B class
- Examples: Mac Studio M4 Max, RTX A6000 48GB, 2x RTX 3090

**Tier 4 — Enthusiast/Server (192GB+ RAM):**
- Models: 70B at Q8, 100B+ models, MoE models
- Examples: Mac Studio M3/M4 Ultra, 4x RTX 3090, H100 80GB

### 6. Deployment Options

**Ollama** (Easiest):
```bash
curl -fsSL https://ollama.com/install.sh | sh
ollama pull phi4-mini
ollama pull gemma3:4b
ollama pull qwen3:4b
```

**llama.cpp** (Most control):
```bash
git clone https://github.com/ggml-org/llama.cpp
cd llama.cpp && make -j
./llama-cli -m model.gguf -p "Hello" -t 8
```

**MLX** (Apple Silicon optimized):
- 20-50% faster than llama.cpp on Apple Silicon
- Unified memory architecture gives Macs unique advantage

**WebLLM** (Browser):
- Runs entirely in browser via WebGPU
- Zero server infrastructure
- Models download once, cache for reuse

### 7. The "4B Class" of 2026

A new category of models around 4B parameters has emerged as the optimal balance of capability and efficiency:

- **Mistral Small 4**: 119B total, 6B active (MoE with 128 experts), 256K context
- **Gemma 4 E2B/E4B**: 2B/4B effective, edge-optimized, multimodal
- **Gemma 4 26B MoE**: 26B total, 3.8B active, high-throughput
- **Phi-4-reasoning-vision**: Mid-fusion architecture, visual + textual reasoning

These models feature configurable reasoning effort, allowing users to toggle between fast responses and deep reasoning.

## Implications for Our Setup

**Current Hardware:** 24GB RAM, Linux x64, no discrete GPU

**Recommended models for our tier:**
- **Qwen3-4B** or **Qwen3-8B** at Q4_K_M — excellent multilingual, dual-mode thinking
- **Phi-4 Mini** (3.8B) — best reasoning at small size
- **Gemma 3 4B** — if multimodal (vision) needed
- **Llama 3.2 3B** — tool calling, structured outputs
- **BitNet 2B** — already working, extremely efficient

**Memory math for our 24GB system:**
- 8B model at Q4_K_M: ~4.7GB weights + ~2GB KV cache + overhead = ~7-8GB total
- 14B model at Q4_K_M: ~8.5GB weights + ~3GB KV cache = ~12-14GB total
- 32B model at Q4_K_M: ~19.5GB weights + ~6GB KV cache = ~26-28GB total (too tight)

**Conclusion:** Our 24GB system is solidly in "Tier 2" — we can comfortably run 8B-14B models at Q4/Q5 quantization. 32B models are possible but tight. BitNet gives us a path to much larger effective models if needed.

## Trends to Watch

1. **MoE (Mixture of Experts) going mainstream** — Mistral Small 4, Gemma 4 26B MoE, Qwen3-30B-A3B. Only a fraction of parameters activate per token, making large models practical.

2. **Knowledge distillation accelerating** — DeepSeek-R1 distillation into 1.5B-8B models. Chain-of-thought reasoning in tiny packages.

3. **1-bit/ternary quantization maturing** — BitNet's 100B-on-CPU demonstration changes what's possible on consumer hardware.

4. **Context windows expanding** — 128K-256K becoming standard even on small models. Long-document RAG and code repository analysis now viable locally.

5. **Multimodal at small scale** — Gemma 3 4B does vision+text on 6GB VRAM. Phi-4-reasoning-vision adds visual reasoning.

6. **"Thinking mode" dual architectures** — Qwen3's toggle between fast direct answers and step-by-step reasoning. Configurable reasoning effort in Mistral Small 4.
