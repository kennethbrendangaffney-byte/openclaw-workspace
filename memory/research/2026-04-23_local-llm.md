## 1. llama.cpp: The Backbone of Local Inference

### Latest Developments (April 2026)
- **Hand-optimized q1_0 dot product kernels** (PR #21636 by pl752): Manual assembly optimization for x86 (AVX2/AVX-512) and generic CPU targets. This targets the most aggressive quantization format (~1.58 bits/weight) and closes the performance gap between local and cloud GPU inference.
- **Gemma 4 audio processing**: llama-server now supports multimodal audio input via Gemma 4's audio conformer encoder. Transforms llama.cpp from text-only to a multimodal platform.
- **Speculative decoding integration**: Works great with Gemma 4 31B — up to 50% speed boost for code generation when paired with a smaller draft model (e.g., Gemma 4 E2B at 4.65B parameters).
- **CPU-only backend improvements**: 64 super-block layout optimizations (commit 36491e4) and MOE GEMV kernel optimizations for batch size > 1 (release b8579).

### Key Insight
Memory bandwidth is the real bottleneck, not raw compute. Quantization shrinks data movement; optimized kernels ensure arithmetic on compressed data doesn't waste cycles. Software improvements often outpace hardware on a cost-per-performance basis.

## 2. BitNet: 1.58-bit Ternary Quantization

### What It Is
Microsoft's open-source inference framework using **ternary weights** (-1, 0, +1). Each weight requires only log₂(3) ≈ **1.58 bits**.

### Why It Matters
- **Massive memory reduction**: 100B model → ~20GB (vs. 50GB for INT4, 200GB for FP16)
- **CPU inference at human reading speed**: 5–7 tokens/sec for 100B models on a single CPU
- **Energy savings**: 55–82% reduction vs. traditional quantization
- **Lookup table (LUT) optimization**: Pre-computed results replace multiply-add operations

### Performance (Real Benchmarks)
| Platform | Speedup | Energy Reduction |
|----------|---------|------------------|
| ARM (Apple M-series, NEON) | 1.37x–5.07x | 55–82% |
| x86 (Intel/AMD, AVX2) | 2.37x–6.17x | 60–82% |

### Kernel Types (Auto-selected)
- **I2_S**: Basic, widest compatibility
- **TL1**: Single-level lookup table
- **TL2**: Two-level lookup table (fastest)

### Available Models
- **BitNet-b1.58-2B-4T** (official, 2B params, 4T tokens, ~400MB RAM)
- Community ports: Llama3-8B-1.58, Falcon3 family

### Limitations
- Models must be **trained with ternary constraints** — can't convert existing FP16 models
- GPU kernels are newer, less optimized than CPU
- Some complex reasoning tasks show slight accuracy degradation

### Comparison Table
| Method | Bits/Weight | Memory (100B) | Accuracy |
|--------|-------------|---------------|----------|
| FP16 | 16 | 200GB | Baseline |
| INT8 | 8 | 100GB | ~99% |
| INT4 (GGUF/GPTQ) | 4 | 50GB | ~94–95% |
| **BitNet b1.58** | **~1.58** | **~20GB** | **~92–96%** |

## 4. GGUF Quantization Best Practices (2026)

### Quality Hierarchy (Best to Worst)
```
32 → 16 → Q8 → Q6_K_M → Q6_K_S → Q6 → Q5_K_M → Q5_K_S → Q5 → Q4_K_M → Q4_K_S → Q4 → Q3_K_M → Q3 → Q2
```

### Practical Selection Strategy
- **Start with Q8 or Q6_K_M** if hardware allows
- **Sweet spot**: Q4_K_M or Q5_K_M for most use cases
- **Memory-constrained**: Q4_K_S or plain Q4
- **Avoid below Q4** unless absolutely necessary — quality degradation becomes visible

### K_M / K_S Suffixes
Mixed quantization variants where important weights keep higher precision. At the same level, Qx_K_M is usually better than plain Qx.

### Rule of Thumb
> "Start with Q8 or Q6_K_M, then move down to Q5 or Q4_K_M only when needed."

## 5. Speculative Decoding: Free Speed Boost

### How It Works
A smaller "draft" model predicts multiple tokens ahead; the larger target model verifies them in parallel. Bypasses the auto-regressive bottleneck.

### Real-World Results (2026)
- **Gemma 4 31B + E2B draft**: 29% average speedup, up to 50% for code generation
- **General expectation**: 20–50% boost depending on task and draft model pairing
- **No quality loss**: Verified tokens are exact; rejected tokens are regenerated

### Best Draft Model Pairings
- Large target (31B) + small draft (2–4B) works best
- Same model family preferred (Gemma 4 + Gemma 4 E2B)
- Draft model should be 5–10x faster than target
