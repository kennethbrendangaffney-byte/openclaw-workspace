## 1. The State of Small Models in 2026

The "small model" space has exploded. We're no longer choosing between "big and slow" vs "tiny and dumb." A 4B model in 2026 punches at the level of a 20–30B model from 2024. Architecture improvements + better training + distillation have rewritten the rules.

### Key Releases (Q1–Q2 2026)

| Model | Params | Size (Q4_K_M) | Notes |
|-------|--------|---------------|-------|
| **Qwen3.5-0.8B** | 0.8B | ~500MB | Multimodal (text/image/video), 262K context, 201 languages |
| **Qwen3.5-2B** | 2B | ~1.3GB | Same architecture as 0.8B, better reasoning |
| **Qwen3.5-4B** | 4B | ~2.5GB | **Beats Qwen3-30B** on long-context tasks. 8GB VRAM target |
| **Qwen3.5-9B** | 9B | ~5.7GB | **Beats gpt-oss-120B** on GPQA Diamond (81.7 vs 80.1) |
| **Ternary Bonsai 8B** | 8B | **1.75GB** | 1.58-bit ternary weights. iPhone-runnable |
| **Ternary Bonsai 4B** | 4B | ~900MB | Same ternary architecture, smaller footprint |
| **Ternary Bonsai 1.7B** | 1.7B | ~400MB | Ultra-edge deployment |
| **MiniCPM4 / 4.1** | ~4B | ~2–2.5GB | "Ultra-efficient LLMs on end devices", 3x speedup on reasoning |
| **Xmodel-2.5** | 1.3B | ~800MB | Data-efficient reasoning SLM |
| **Nanbeige4.1-3B** | 3B | ~1.8GB | Strong agentic behavior, code gen, general reasoning |

### The Big Shift

- **Qwen3.5-9B outperforms a 120B model** on graduate-level reasoning. Parameter count is no longer the proxy for capability.
- **Gated DeltaNet hybrid attention** (3 linear attention blocks : 1 softmax block) keeps memory near-linear with sequence length — making 262K context feasible even on 0.8B models.
- **Multi-token prediction** is now standard in small models, boosting inference throughput.

## 2. Quantization: The Practical Guide

### GGUF Quantization Levels (2026 Best Practice)

| Level | Bits/Weight | Size vs FP16 | Use Case | Quality Loss |
|-------|-------------|--------------|----------|--------------|
| **Q2_K** | ~2.5 | ~20% | Extreme compression, edge devices | Noticeable |
| **Q3_K_M** | ~3.5 | ~25% | Low-VRAM scenarios | Moderate |
| **Q4_K_M** | ~4.5 | ~30% | **The sweet spot** — general use | Minimal |
| **Q5_K_M** | ~5.5 | ~35% | Complex reasoning, extra VRAM | Very low |
| **Q6_K** | ~6.0 | ~38% | High-quality local inference | Negligible |
| **Q8_0** | 8.0 | ~50% | Maximum quality, abundant memory | None perceptible |
| **FP16/BF16** | 16.0 | 100% | Training, critical accuracy | None |

### Rules of Thumb

- **Always prefer K-quants over legacy `_0` formats.** K-quants use k-means clustering and preserve more information.
- **Q4_K_M is the default choice** for 95% of use cases. The quality loss is barely perceptible for code gen, writing, Q&A, summarization.
- **Q5_K_M** when you have the VRAM and need complex reasoning (math, logic, coding).
- **Q8_0** only when accuracy is critical and memory is abundant.
- **CPU-only inference:** Q4_K_M is optimal. Higher quants don't meaningfully improve CPU throughput because memory bandwidth, not precision, is the bottleneck.

### Dynamic Quantization (Unsloth Dynamic 2.0)

Unsloth's approach **upcasts important layers to higher precision** while keeping less critical layers compressed. This gives better quality at smaller sizes than uniform quantization. Already available for Qwen3.5 models on HuggingFace.

## 3. llama.cpp: The CPU Inference King

### Why llama.cpp Dominates Local Inference

- **Pure C/C++ with GGML backend** — zero Python overhead, maximum portability
- **105K+ GitHub stars** — the de facto standard for open-weight local runtime
- **Cross-platform:** x86, ARM, RISC-V, PowerPC, Apple Silicon
- **Quantization-native:** Built around GGUF from day one

### 2026 Developments

#### Low-Level CPU Optimizations (April 2026)

PR #21636 by contributor pl752: **"ggml-cpu: Optimized x86 and generic cpu q1_0 dot (follow up)"**

- Hand-optimized dot product operations for **q1_0 quantization** (~1.58 bits/weight)
- Uses **AVX2/AVX-512** vector instructions on x86, with generic CPU fallbacks
- The memory bandwidth savings from aggressive quantization + optimized arithmetic = throughput improvements no CPU upgrade could match
- This is the kind of "unglamorous" engineering that quietly moves the practical frontier

#### RISC-V & PowerPC Support (Feb–Apr 2026)

- **b8099 (Feb 2026):** PowerPC FP16 MMA path added
- **b8814 (Apr 2026):** 128-bit RVV (RISC-V Vector) implementation
- llama.cpp is becoming truly architecture-agnostic

#### 64 Super-Block Layout (Feb 2026)

Commit `36491e4` — CPU-only backend optimization for 64 super-block layout. Improves cache locality and reduces memory bandwidth pressure.

### Running Models with llama.cpp

```bash
# Basic CLI inference
llama-cli -m ./models/Qwen3.5-4B-Q4_K_M.gguf \
  -p "Explain quantum computing" -n 512 --ctx-size 4096

# OpenAI-compatible server
llama-server -m ./models/Qwen3.5-4B-Q4_K_M.gguf \
  --port 8080 --ctx-size 4096

# Optimal CPU flags
llama-cli -m model.gguf -t $(nproc) --ctx-size 4096 \
  --batch-size 512 --ubatch-size 512
```

## 4. BitNet & Ternary Quantization

### The 1.58-Bit Revolution

BitNet b1.58 uses **ternary weights: -1, 0, +1**. That's ~1.58 bits of information per weight (log₂(3) ≈ 1.585). The theoretical promise: massive compression with minimal quality loss.

### Key Research

- **"BitNet b1.58 Reloaded"** (July 2024, arXiv:2407.09527): Demonstrated SOTA performance on smaller networks, not just large ones.
- **"When are 1.58 bits enough?"** (Nov 2024, arXiv:2411.05882): Bottom-up exploration of when ternary quantization is sufficient. Found it works surprisingly well for many tasks.
- **Microsoft open-sourced BitNet** enabling 100B parameter inference on a single CPU.

### Practical Implementations

#### Microsoft's BitNet (Official)

- **BitNet-b1.58-2B-4T** — 1.3GB download, 1.1GB RAM at runtime
- **~27 tokens/sec** on 8-thread CPU (I2_S format)
- **Limitation:** 2B is the largest reliable model currently. 8B models exist but have conversion issues.
- **Context:** 4096 tokens

#### PrismML's Bonsai Series (April 2026)

PrismML is doing the most practical ternary work right now:

| Model | Params | Memory | Architecture |
|-------|--------|--------|--------------|
| 1-bit Bonsai 8B | 8B | **1.15GB** | Binary (-1, +1), designed from scratch |
| Ternary Bonsai 8B | 8B | **1.75GB** | Ternary (-1, 0, +1), better quality |
| Ternary Bonsai 4B | 4B | ~900MB | Same ternary design |
| Ternary Bonsai 1.7B | 1.7B | ~400MB | Ultra-compact |

**Critical insight:** PrismML designs models **from the ground up** for ternary weights, rather than quantizing post-hoc. This avoids the quality degradation that comes from forcing a 16-bit model into 1.58 bits.

#### Hugging Face Transformers Support

- Transformers v4.57.0 includes **native BitNet support**
- Replaces linear layers with `BitLinear` layers
- Weights: ternary (-1, 0, +1), Activations: 8-bit

### The Tradeoff

| Approach | Size Reduction | Quality | Availability |
|----------|---------------|---------|--------------|
| Standard GGUF Q4_K_M | ~70% | Excellent | Universal |
| BitNet 2B (I2_S) | ~90% | Good | Limited models |
| 1-bit Bonsai 8B | ~93% | Good | PrismML only |
| Ternary Bonsai 8B | ~89% | Better | PrismML only |

**Verdict:** For most practical use in 2026, Q4_K_M GGUF on a well-architected small model (Qwen3.5-4B, MiniCPM4) gives the best quality/size tradeoff. BitNet/ternary is promising for extreme edge cases but the model ecosystem is still limited.
