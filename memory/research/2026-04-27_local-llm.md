## 1. The State of Local LLM Inference in 2026

Local LLM inference has moved from experimental to genuinely viable. Key enablers:

- **llama.cpp** — the de facto standard for local inference. C/C++ engine, minimal setup, wide hardware support (CPU, GPU, Apple Silicon, ARM).
- **Ollama** — easiest entry point. Wraps llama.cpp with a simple CLI and model registry.
- **MLX** — Apple Silicon optimized, 20–50% faster than llama.cpp on M-series chips.
- **WebLLM** — runs entirely in browser via WebGPU, no server needed.

The bottleneck is no longer raw compute (FLOPS) but **memory bandwidth and VRAM capacity**. Each token requires reading the full model state, so bandwidth dominates throughput.

## 3. Quantization: Making Models Fit

**GGUF** (introduced by llama.cpp) is the de facto standard for local inference. It reduces weight precision from 32-bit floats to 4–8-bit integers, cutting memory usage by 2–8x.

### Quantization Levels

| Level | Quality | Size Reduction | Use Case |
|-------|---------|----------------|----------|
| Q8_0 | Minimal loss | 2x | Quality-critical |
| Q5_K_M | Low loss | 3x | **Best balance** |
| Q4_K_M | Low-moderate | 4x | Limited VRAM |
| Q3_K_M | Moderate | 5x | Memory-critical |
| Q2_K | Noticeable | 8x | Last resort |

### Key Formats

- **K-Quants** (Q4_K_M, Q5_K_M) — Most common. Good quality/size tradeoff.
- **I-Quants** (IQ4_XS, etc.) — Newer, often better quality at same size.
- **Imatrix quantization** — Uses importance matrix for better quality. Recommended for production.

### Imatrix Quantization (Best Practice)

```bash
# Generate importance matrix
./llama-imatrix -m model.gguf -f calibration_data.txt -o imatrix.dat

# Quantize with imatrix
./llama-quantize --imatrix imatrix.dat model.gguf model-q4.gguf Q4_K_M
```

### Hardware Requirements (Quantized)

| Model Size | FP16 | Q4 Quantized | Recommended GPU |
|------------|------|--------------|-----------------|
| 1–2B | 2–4 GB | 1–2 GB | Any 4GB+ GPU |
| 3–4B | 6–8 GB | 2–4 GB | RTX 3060 |
| 7B | 14–16 GB | 3.5–5 GB | RTX 3060 12GB |
| 13–14B | 26–28 GB | 8–10 GB | RTX 4090 |

## 4. BitNet: The 1.58-Bit Revolution

**BitNet b1.58** from Microsoft Research uses **ternary quantization** — weights are only -1, 0, or +1. This is ~1.58 bits per weight (hence the name).

### How It Works

- Eliminates floating-point operations
- Replaces them with simple integer additions
- Compresses each parameter to ~0.2 bytes
- Uses custom kernels (bitnet.cpp) for fast CPU inference

### Performance Claims

- **100B parameter model** runs at human reading speed (5–7 tok/s) on a single CPU
- **Up to 96% less energy** than full-precision equivalents
- **Lossless inference** — no quality degradation vs full-precision BitNet models

### Available Models

| Model | Size | Training | Status |
|-------|------|----------|--------|
| BitNet b1.58 2B-4T | 1.3GB download | 4T tokens | ✅ Released |
| BitNet b1.58 8B | Larger | - | ⚠️ Requires TL1 kernel rebuild |
| BitNet b1.58 100B | ~20GB | - | 🧪 Research demo |

### Our Experience

- **Working model:** BitNet b1.58-2B-4T (I2_S format)
- **Speed:** ~27 tok/s on CPU (8 threads)
- **RAM:** 1.1GB at runtime
- **Limitation:** 2B is the largest reliable model currently. 8B models have kernel issues.

### BitNet vs Standard Quantization

| Aspect | BitNet | GGUF Q4 |
|--------|--------|---------|
| Bits/weight | 1.58 | 4 |
| Operations | Integer add | Integer multiply-add |
| Speed (CPU) | Very fast | Moderate |
| Model availability | Limited (Microsoft only) | Universal |
| Ecosystem | Emerging (bitnet.cpp) | Mature (llama.cpp/Ollama) |

## 5. llama.cpp Server: OpenAI-Compatible API

llama.cpp now includes `llama-server` — an HTTP server with OpenAI-compatible endpoints.

### Key Features

- **OpenAI API compatibility** — drop-in replacement for GPT-4 endpoints
- **Chat completions** — `/v1/chat/completions`
- **Text completions** — `/v1/completions`
- **Embeddings** — `/v1/embeddings`
- **Multi-model support** — load multiple models simultaneously

### Basic Usage

```bash
# Build with server support
git clone https://github.com/ggml-org/llama.cpp
cd llama.cpp && cmake -B build && cmake --build build --config Release

# Run server
./build/bin/llama-server -m model.gguf -c 4096 --port 8080

# Query with curl
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "default",
    "messages": [{"role": "user", "content": "Hello!"}]
  }'
```

### Advantages Over Ollama

- More control over inference parameters
- Better for production deployments
- Direct access to llama.cpp optimizations
- Lower overhead than Ollama wrapper
