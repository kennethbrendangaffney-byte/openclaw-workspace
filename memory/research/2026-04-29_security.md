## 1. 2026 Cybersecurity Threat Landscape

### Key Trends
- **AI-assisted attacks**: Reconnaissance and exploitation cycles now run in hours, not weeks. Attackers use automation to scan for exposed services and known CVEs at scale.
- **Living off the land**: 79% of Linux attacks use no malware whatsoever — attackers log in with stolen credentials and use built-in tools (bash, cron, curl, wget). Signature-based antivirus is largely irrelevant.
- **Supply chain compromise**: Poisoned dependencies, libraries, and build tools spread malware through normal update channels. The victim "updates software" rather than "downloads malware."
- **Ransomware targeting servers**: Server-side ransomware is lucrative. Encrypting a database or production filesystem stops operations cold.
- **Identity-centric operations**: PwC threat intelligence expects 2026 to be defined by stealthier, persistent, identity-focused attacks connected to geopolitical conflicts.

### Linux-Specific Threats
- **SSH brute force & credential stuffing**: 89% of Linux endpoint attacks involve this. A fresh VPS receives its first probing attempt within 90 seconds of going live.
- **Privilege escalation CVEs**: Linux's privilege model is a strength, but when escalation vulnerabilities surface, they're catastrophic. 5,530+ kernel CVEs were logged in 2025.
- **Cryptojacking**: Compromised Linux nodes are commonly used for crypto mining, DDoS, or as infrastructure for further intrusions.
- **Kernel-level rootkits**: Advanced threats that operate below the application layer.

### The "Linux Is Safe" Myth
Linux resists Windows-style viruses by design (no auto-executing binaries, strict privilege separation, rapid patching). But "virus-resistant" is not "attack-proof." The real threats exploit misconfigurations, reused passwords, unpatched kernels, and stolen credentials — none of which antivirus catches.

## 2. Linux Hardening Techniques (2026 Best Practices)

### Core Philosophy: Assume Breach, Reduce Exposure
The goal isn't an unattackable system — it's a system where an attacker who gets in can't do much, can't stay long, and can't hide quietly.

### OS Hardening Checklist
1. **Minimal install**: Every extra package is one more thing to patch. Install only what's needed.
2. **Service lockdown**:
   - Don't bind services to all interfaces if they only need local access
   - Don't run services as root if they don't need it
   - Disable unused services
3. **File permissions**: Audit world-writable directories, sloppy config files, permissive home folders
4. **Mandatory Access Control (MAC)**:
   - **AppArmor** (Ubuntu default) or **SELinux** (RHEL/CentOS)
   - Can contain damage even when a service is exploited
   - In one finance environment, SELinux prevented attackers from pivoting after compromising a web service

### SSH Hardening (Critical)
- Disable password authentication → switch to key-based login only
- Change default port (22) to something non-standard
- Use `fail2ban` or `CrowdSec` to block brute-force attempts
- Restrict source IPs where possible
- Enable MFA if feasible
- Keep OpenSSH updated (CVE-2025-26465 showed SSH itself can be weaponized)

### Patching & Vulnerability Management
- Treat patching as a pipeline: discover → prioritize → test → deploy → verify
- Critical security fixes need hours, not weeks
- Use vulnerability scanners to catch misconfigurations, weak SSH settings, exposed admin panels
- Monitor for **configuration drift** — temporary changes that never get reverted
- Track third-party dependencies separately from OS patches

### Network Segmentation
- Public-facing services in DMZ
- App tiers in protected internal zones
- Databases in heavily restricted networks
- Management interfaces isolated and locked down
- Default-deny host firewalls with explicit allow rules only

### Kernel-Level Protections
- Enable kernel security features (sysctl hardening)
- Keep kernels patched against privilege escalation CVEs
- Consider `auditd` for system call auditing
- Use `Lynis` for automated security auditing

## 3. Privacy Tools & Self-Hosted Security Stack

### Essential Self-Hosted Security Tools (2026)

| Layer | Tool | Purpose |
|-------|------|---------|
| **Network perimeter** | **WG-Easy** (WireGuard VPN) | Keep admin services off public internet |
| **Intrusion prevention** | **CrowdSec** + **Fail2Ban** | Block brute force, scans, known malicious IPs |
| **Secrets management** | **HashiCorp Vault** / **OpenBao** | Centralized encrypted credential storage |
| **Password management** | **Vaultwarden** | Self-hosted Bitwarden-compatible password vault |
| **SSL/TLS** | **Let's Encrypt** (certbot) | Automated certificate management |

#### CrowdSec (Modern Alternative to Fail2Ban)
- 60x faster than Fail2Ban (written in Go)
- IPv6 compatible, built-in WAF component
- Community blocklist updated by thousands of nodes
- Shares threat intelligence across the network
- Works with Docker, Kubernetes, Linux, OPNsense

#### WG-Easy (WireGuard VPN)
- Web UI for managing WireGuard — no command-line gymnastics
- One-click client creation with QR codes
- Access self-hosted services securely from anywhere
- Keep admin panels (Portainer, DB dashboards) off public internet
- Lightweight, runs on minimal resources

#### Vaultwarden (Password Management)
- Full Bitwarden compatibility — no vendor lock-in
- AES-256 encryption with Argon2 key derivation
- Uses ~50MB RAM, runs on Raspberry Pi
- Supports organizations, shared vaults, 2FA

#### HashiCorp Vault / OpenBao (Secrets Management)
- Centralized encrypted store for API keys, DB passwords, TLS certs, SSH credentials
- Dynamic secrets (auto-generated, auto-expired database credentials)
- **OpenBao**: Linux Foundation fork of Vault's last truly open-source version (MIT license)

### Additional Privacy-Focused Self-Hosted Apps
- **Nextcloud**: File storage, sync, calendars, contacts, document collaboration
- **Pi-hole** / **AdGuard Home**: Network-wide ad blocking
- **Bitwarden** (official) / **Vaultwarden**: Password management
- **Mastodon**: Decentralized social networking
- **Ghost**: Clean blogging/publishing platform
- **Beszel**: Lightweight server and Docker monitoring
- **Homepage**: Dashboard for all self-hosted services

### Defense in Depth Principle
No single tool is enough. Each layer catches what others miss:
1. VPN for admin access
2. CrowdSec + Fail2Ban for perimeter defense
3. Vault/OpenBao for secrets
4. Vaultwarden for passwords
5. Let's Encrypt for encryption in transit

## 4. Personal OPSEC (Operational Security)

### Threat Modeling for Individuals
1. **Identify your digital assets**: Accounts, devices, data, financial info, communications
2. **Assess threats**: What are you protecting against? (identity theft, doxxing, surveillance, targeted attacks)
3. **Evaluate vulnerabilities**: Weak passwords, reused credentials, unpatched devices, oversharing online
4. **Develop mitigations**: Layered defenses based on your actual threat model

### Practical OPSEC Checklist
- **Use a password manager** (Vaultwarden/Bitwarden) — unique, strong passwords for every service
- **Enable 2FA/MFA everywhere possible** — prefer authenticator apps or hardware keys over SMS
- **Encrypt your devices** — full-disk encryption on laptops, phones
- **Use a VPN** for untrusted networks (WireGuard self-hosted or Mullvad/ProtonVPN)
- **Separate work/personal accounts** — compartmentalization limits blast radius
- **Review app permissions** regularly — revoke unnecessary access
- **Use aliases for online accounts** — SimpleLogin, AnonAddy for email aliases
- **Minimize data sharing** — question why every service needs your real info
- **Regular backups** — encrypted, tested, offsite or self-hosted (Nextcloud)
- **Keep software updated** — OS, apps, browsers, extensions
- **Be cautious with public WiFi** — assume it's monitored
- **Review your digital footprint** periodically — search yourself, clean up old accounts

### Linux Workstation Specifics
- **Firewall**: Enable UFW (Uncomplicated Firewall) — `sudo ufw enable`
- **Automatic updates**: Configure unattended-upgrades for security patches
- **AppArmor**: Verify it's enforcing — `sudo aa-status`
- **Audit with Lynis**: `sudo lynis audit system` — gives a hardening index and actionable recommendations
- **Check for exposed services**: `sudo ss -tlnp` — see what's listening
- **Review sudo access**: `sudo cat /etc/sudoers` and `/etc/sudoers.d/*`
- **Encrypt home directory** or full disk if not already done
- **Use Flatpak/Snap sandboxing** where available for untrusted applications

### Emergency Protocol (24-Hour Digital Disappearance)
If you need to rapidly reduce your digital exposure:
1. Kill "hot" accounts (social media, public profiles)
2. Mask physical location via metadata scrubbing
3. Freeze financial trails (credit freeze, notify banks)
4. Deploy a "digital smoke screen" — aliases, temporary accounts, disinformation
5. Secure comms — switch to encrypted, ephemeral messaging
