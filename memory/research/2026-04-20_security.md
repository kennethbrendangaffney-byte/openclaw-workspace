## ⚠️ Current Threat Landscape (2026)

### Critical Vulnerabilities & Active Threats

**1. AI-Enhanced Social Engineering**
- Deepfake voice/video calls for financial fraud and identity theft
- LLM-powered phishing with perfect grammar and contextual awareness
- Real-time translation scams targeting multilingual organizations
- **Mitigation:** Verify out-of-band for sensitive requests, establish verbal codewords

**2. Supply Chain Attacks**
- Compromised open-source dependencies and npm/PyPI packages
- Malicious VS Code extensions and developer tooling
- CI/CD pipeline poisoning via compromised build agents
- **Mitigation:** Pin dependencies, verify checksums, isolated build environments

**3. Post-Quantum Cryptography Transition**
- NIST finalized PQC standards (CRYSTALS-Kyber, Dilithium, SPHINCS+)
- "Harvest now, decrypt later" threat for current encrypted data
- Transition period creating hybrid classical/quantum vulnerabilities
- **Mitigation:** Begin inventorying crypto usage, test PQC libraries

**4. Ransomware Evolution**
- Triple extortion: encrypt + exfiltrate + DDoS + regulatory threats
- Targeting Linux/ESXi systems specifically (previously Windows-focused)
- Initial access via VPN appliances and edge devices
- **Mitigation:** Offline backups, network segmentation, patch edge devices first

**5. IoT and Edge Device Botnets**
- Massive botnets exploiting default credentials in smart devices
- Used for DDoS and crypto mining
- **Mitigation:** Change defaults, isolate IoT networks, disable UPnP

## 🛡️ Linux Hardening Techniques

### Essential Practices

**1. Minimal Attack Surface**
- Install only necessary packages (`apt autoremove` regularly)
- Disable unused services: `systemctl list-units --type=service --state=running`
- Remove unnecessary users and set strong password policies
- Use `chkrootkit` and `rkhunter` periodically

**2. Firewall & Network Security**
- **UFW (Uncomplicated Firewall):** `ufw default deny incoming`, allow only specific ports
- **iptables/nftables:** For granular control
- Disable IPv6 if not needed (reduces attack surface)
- Network segmentation: isolate services with VLANs or containers

**3. SSH Hardening**
- Disable root login: `PermitRootLogin no`
- Key-based auth only, disable password auth
- Change default port (security through obscurity + log noise reduction)
- Use `fail2ban` to block brute-force attempts
- `AllowUsers` or `AllowGroups` to restrict access
- Consider `WireGuard` or `Tailscale` instead of exposed SSH

**4. File System Security**
- Separate partitions for `/tmp`, `/var`, `/home` with `noexec`, `nosuid`, `nodev`
- Enable `AIDE` or `Tripwire` for file integrity monitoring
- Use `LUKS` for full-disk encryption
- Set up `auditd` for system call auditing

**5. Kernel Hardening**
- Enable `SELinux` or `AppArmor` (mandatory access control)
- `sysctl` hardening:
  - `kernel.unprivileged_bpf_disabled=1`
  - `kernel.kptr_restrict=2`
  - `kernel.dmesg_restrict=1`
  - `net.ipv4.conf.all.rp_filter=1` (reverse path filtering)
- Consider `grsecurity` patches (if available for your kernel)

**6. Container & Application Security**
- Run services as non-root users
- Use `systemd` sandboxing features (`PrivateTmp`, `NoNewPrivileges`, `ProtectSystem`)
- Container runtime security: `seccomp`, `AppArmor` profiles
- Regular vulnerability scanning: `lynis`, `OpenSCAP`

## 🕵️ Personal OPSEC (Operational Security)

### Digital Identity Compartmentalization

1. **Separate Personas**
   - Different email addresses for different contexts (work, personal, shopping, sensitive)
   - Unique usernames where possible; avoid linking identities
   - Consider separate devices or VMs for high-risk activities

2. **Communication Security**
   - Prefer Signal, Matrix, or Session for sensitive comms
   - Verify safety numbers/out-of-band keys
   - Assume all unencrypted communication is public

3. **Financial Privacy**
   - Use privacy-focused cards or virtual cards for online purchases
   - Consider Monero for cryptocurrency privacy (Bitcoin is pseudonymous, not anonymous)
   - Separate financial accounts by purpose

4. **Physical Security**
   - Full-disk encryption on all devices (LUKS, FileVault, BitLocker)
   - BIOS/UEFI passwords
   - Tamper-evident seals on critical hardware
   - Consider Faraday bags for sensitive travel

5. **Social Engineering Defense**
   - Establish family/friend verification protocols
   - Be suspicious of urgency and authority-based pressure
   - Never give 2FA codes to anyone, ever
   - Verify unexpected requests through alternate channels

### Threat Modeling Framework

1. **What do you want to protect?** (assets)
2. **Who do you want to protect it from?** (adversaries)
3. **How likely is an attack?** (risk assessment)
4. **What are the consequences of failure?** (impact)
5. **How much effort are you willing to expend?** (cost/benefit)

**Remember:** Perfect security is impossible. Goal is to raise the cost of attack above the value of the target.

## 🏠 Self-Hosted Security Stack (Recommended)

### Minimal Viable Secure Setup

```
Router/Firewall
├── Pi-hole/AdGuard Home (DNS filtering)
├── WireGuard/Headscale (remote access)
└── DMZ
    ├── Reverse Proxy (Traefik/NPM with Authelia)
    ├── Nextcloud (files, calendar, contacts)
    ├── Vaultwarden (passwords)
    ├── Mail server (if needed — complex)
    └── Monitoring (Uptime Kuma, Netdata)
```

### Key Principles
- **Defense in depth:** Multiple overlapping security controls
- **Least privilege:** Every component has minimum necessary access
- **Assume breach:** Design for containment, not just prevention
- **Automated backups:** 3-2-1 rule (3 copies, 2 media, 1 offsite)
- **Regular updates:** Automate where possible, test before applying
